The public demo should prove that a personal site can read serious Org, not only
three TODO rows.  This source intentionally mixes prose, tables, links, drawers,
timestamps, source blocks, lists, footnotes, and planning metadata.

The key rule is simple: TypeScript may arrange the experience, but native Org
syntax remains the source of truth.[fn:native-org]

** What the reader should notice

- *Blog* headings still become articles when tagged =:blog:=.
- /Italic/, *bold*, _underline_, +strike-through+, =verbatim=, and ~code~ should
  survive rendering.
- Links can point to [[https://orgmode.org/][Org mode]], [[id:syntax-table-record][stable records]],
  and [[attachment:syntax-map.png][attachment-like evidence]].
- Footnotes, tables, drawers, and source blocks should not flatten into plain
  text.

#+BEGIN_QUOTE
Knowledge is useful only when it can be revisited, re-ranked, and acted on.
#+END_QUOTE

[fn:native-org] In this demo that means property drawers, planning lines,
headline tags, source blocks, and rendered HTML all come through the WASM parser
boundary instead of a TypeScript approximation.

** Inline and block syntax

1. Inline emphasis: *bold*, /italic/, _underline_, +deleted+, =verbatim=,
   ~inline-code~.
2. Nested list:
   - Parser-owned facts
     - TODO keyword
     - priority cookie
     - planning line
   - Frontend-owned presentation
     - source cards
     - article library
     - agenda panels
3. Checkbox state:
   - [X] render checked task
   - [-] render partial task
   - [ ] render open task
